# Dugri — Onboarding & Legal Spec

**Status:** Requirements & Features Document — to be handed to Claude Code for implementation planning
**Stack:** Flask 3.1, MongoDB, Jinja2, Python 3.12, existing `dashboard/` service on Railway
**Language:** Spec in English; user-facing copy in Hebrew (provided in section 9)

---

## 1. Context & Goals

Dugri is a Telegram bot that helps Israelis track healthy habits — meals (calories + protein), eating window, sleep, workouts, and weekly self-care. The actual product experience lives inside Telegram. The website exists for three reasons only:

1. **Account creation** — authenticate the user and create their record in MongoDB
2. **Legal consent** — capture explicit, documented consent to terms of service and privacy policy (regulatory requirement under Israeli privacy law)
3. **Bridge to Telegram** — open the bot with a deep-linked session token so the bot can recognize the user

Everything else — getting to know the user, profile setup, habit selection, goals, motivation — happens inside the bot, conversationally, after the user is already there. This is a foundational design choice: the website doesn't pretend to be the product.

The onboarding goal is **time from landing-page-arrival to first-meal-logged-in-bot under 3 minutes**, with no friction beyond what's legally required.

## 3. Authentication Flow

**Mechanism:** Google OAuth via Authlib (already in the dashboard service). No alternative login methods at MVP.

**Data captured from Google:** Email (required), full name (required), profile photo URL (optional, stored if present for future personalization in the bot).

**Behavior:**

1. User clicks "התחבר עם Google" → standard Google OAuth flow opens
2. Google redirects back to the callback route with auth code
3. Backend validates the code, retrieves email + name + photo
4. **Before creating the user record**, backend verifies that checkboxes 1 and 2 were submitted (the form sends them along with the OAuth state)
5. If consent missing → redirect back with terracotta-colored error: "נדרש לאשר את תנאי השימוש כדי להמשיך"
6. If consent present → create `user_profiles` document with:
   - `email`, `name`, `photo_url` (if any)
   - `consents`: object with `terms_accepted_at`, `privacy_accepted_at`, `medical_disclaimer_accepted_at`, `marketing_opt_in` (bool with timestamp if true)
   - `created_at`, `updated_at`
   - `telegram_user_id`: null (filled later when user opens the bot)
   - `signup_session_token`: a random 32-char token (used in the Telegram deep link)
   - `trial_started_at`: null (set when user first interacts with the bot, not at signup)
   - `subscription_status`: "trial_pending"
7. Render the post-signup zone with the "Open Dugri in Telegram" button

**Session handling:** Standard Flask session cookie (already configured). Keep the user logged in for 30 days so returning users see the dashboard or "open bot" button instead of repeating the flow.

**Returning users (already signed up):** If a user with this email already exists, skip the consent capture (their previous consent is on file) and go directly to the post-signup zone. If their `telegram_user_id` is already set, redirect to `/dashboard` instead.

## 4. Telegram Linking — Deep Link Mechanism

**Approach:** Token-based deep link. No phone number collection. This is industry standard and removes friction.

**Flow:**

1. After successful signup, the website generates a `signup_session_token` (32-char random string) and stores it on the user record
2. The "Open Dugri in Telegram" button opens `https://t.me/<DUGRI_BOT_USERNAME>?start=<signup_session_token>`
3. Telegram opens the bot and sends `/start <signup_session_token>` as the user's first message
4. The bot receives this in its start handler:
   - Parse the token from the start payload
   - Look up the `user_profiles` document with that `signup_session_token`
   - If found and `telegram_user_id` is null: link the Telegram user ID to that record, clear the token (single-use), set `trial_started_at = now()` and `subscription_status = "trial_active"`
   - If found but `telegram_user_id` already set: ignore the token (account already linked), greet normally
   - If not found (expired/invalid token): respond with "נראה שהקישור פג. חזור לאתר ונסה שוב: [URL]"
5. Once linked, the bot starts the conversational onboarding (out of scope for this spec — handled separately in the bot service)

**Token security:**
- Tokens are cryptographically random (use `secrets.token_urlsafe(24)`)
- Tokens expire 24 hours after creation. If a user clicks the link after expiration, the bot tells them to return to the website. The website should regenerate a fresh token on every visit by a logged-in-but-not-yet-linked user.
- Tokens are single-use — once consumed by the bot, they're deleted from the record

**Edge case — user closes the tab before opening Telegram:** When they return to the website (even days later) while still in their browser session, they should land on the post-signup zone again with the same button, regenerated token. If their session expired, they re-do Google OAuth (which is instant if they're still logged into Google) and skip the consent step.

## 5. Trial & Subscription Model (V1)

**21-day free trial**, no credit card required at signup. After day 21:

- The bot sends a message: "השבועיים וחצי שלך עם דוגרי הסתיימו. אם בא לך להמשיך — זה 47 ₪ בחודש. רוצה? [כפתור: כן, להמשיך]"
- Tapping the button opens a payment page on the website (out of scope for V1 implementation — placeholder OK for MVP)
- If the user doesn't pay, the bot enters a degraded state: it still responds, but only with "התקופה שלך הסתיימה. אם תחליט להמשיך, [קישור]" — no meal logging, no insights
- Users who don't pay remain in the database; their data isn't deleted unless they actively request it

**No automatic charging** at end of trial. The user must take an explicit action to subscribe. This decision is intentional and brand-consistent: Dugri doesn't surprise you.

**Implementation scope for this spec:** Track `trial_started_at` and `subscription_status` on the user profile. Build the trial-end notification mechanism in the bot (separate spec). Payment page is a placeholder route that returns "בקרוב" for now.

## 8. Technical Implementation Notes

**Existing assets to reuse:**
- Google OAuth setup (Authlib) — already configured for the existing dashboard
- MongoDB connection and `user_profiles` collection — extend with new fields
- Jinja2 template base — extend with the cream/cinnamon design tokens
- Existing Hebrew/RTL CSS conventions

**New routes:**

| Route | Method | Purpose |
|---|---|---|
| `/` | GET | Onboarding landing page (this spec) |
| `/auth/google/start` | GET | Initiates Google OAuth (existing) |
| `/auth/google/callback` | GET | OAuth callback, captures consent, creates user (existing — extend with consent capture) |
| `/terms` | GET | Renders terms of service from a markdown or template file |
| `/privacy` | GET | Renders privacy policy from a markdown or template file |
| `/about` | GET | Founder bio page (placeholder OK for V1) |
| `/api/regenerate-bot-link` | POST | For returning logged-in users — generates a fresh session token and returns the deep link URL |

**New env variables:**

| Variable | Purpose |
|---|---|
| `DUGRI_BOT_USERNAME` | Telegram username of the bot (used in deep link URL) |
| `FOUNDER_TELEGRAM_USERNAME` | Founder's personal Telegram username (used by admin dashboard outreach) |
| `CONTACT_EMAIL` | Displayed in terms and privacy |
| `MARKETING_FROM_EMAIL` | Sender address for marketing emails (only used in V2) |

**New / extended MongoDB fields on `user_profiles`:**

```
{
  "_id": ObjectId,
  "email": str,
  "name": str,
  "photo_url": str | null,
  "telegram_user_id": int | null,
  "signup_session_token": str | null,
  "signup_session_token_expires_at": datetime | null,
  "consents": {
    "terms_accepted_at": datetime,
    "privacy_accepted_at": datetime,
    "medical_disclaimer_accepted_at": datetime,
    "marketing_opt_in": bool,
    "marketing_opt_in_at": datetime | null,
    "consent_version": str  // e.g. "2026-05-19" — bump if terms change so we can re-prompt
  },
  "trial_started_at": datetime | null,
  "subscription_status": "trial_pending" | "trial_active" | "trial_ended" | "paid" | "cancelled",
  "created_at": datetime,
  "updated_at": datetime
}
```

**Tests to write:**
- OAuth callback with all consents → user created correctly
- OAuth callback missing required consent → redirected with error, no user created
- Returning user with active session → skips consent step
- Telegram token consumed by bot → user linked, token cleared, trial started
- Telegram token expired → bot returns user to website with clear message
- Marketing opt-in toggle → stored correctly
